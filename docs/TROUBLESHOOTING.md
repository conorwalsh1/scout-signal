# Scout Signal — Troubleshooting

Runbooks for common production and deployment issues.

---

## CloudFront 403 on production (login/signup or forms fail)

### Symptom

- Users see: **"The request could not be satisfied. This distribution is not configured to allow the HTTP request method that was used for this request. The distribution supports only cachable requests. HTTP ERROR 403"**
- The error page says **"Generated by cloudfront (CloudFront)"**.
- GET requests (e.g. loading the homepage) may work; **POST** requests (login, signup, any form submit) return 403.

### Cause

Traffic for your domain is reaching **AWS CloudFront** instead of **Vercel**. This project runs on Vercel; CloudFront should not be in the path. CloudFront distributions are often configured to allow only cacheable methods (GET, HEAD), so POST is rejected with 403.

So somewhere, **DNS or routing is still sending traffic to CloudFront** (e.g. a CNAME or A record pointing to a CloudFront hostname or an old AWS setup).

### Fix: Point the domain only to Vercel

For **signalscoutradar.com**, DNS must point **only** to Vercel. If the domain is managed in Vercel (bought there, nameservers = Vercel), use:

- **Vercel Dashboard** → **scout-signal** project → **Settings** → **Domains**  
  or top-level **Domains** → open **signalscoutradar.com**
- Open the **DNS Records** section for **signalscoutradar.com**.

**Target DNS state:**

| Name | Type | Value |
|------|------|--------|
| `@` | A | `76.76.21.21` |
| `www` | CNAME | `cname.vercel-dns.com` |

Use the exact values Vercel shows under **"DNS Change Recommended"** for this project if they differ from the table above.

- **Remove** any record that points to a hostname ending in **cloudfront.net** or to an IP you know is AWS.
- **Remove** any old `www` A record if one exists.
- **Remove** any URL forwarding, masked forwarding, or parking rule for either `signalscoutradar.com` or `www.signalscoutradar.com`.
- Ensure the **root** (`@`) uses the A record above and **www** uses the CNAME above (or Vercel’s current recommended values).

### If Vercel’s "Edit" fails with validation errors

When editing an A or CNAME record, Vercel sometimes shows **"Invalid request: 'value' should match format 'ipv4'"** or **"Invalid \`name\` parameter."** Workarounds:

1. **Add new, then delete old**
   - **Add** a new A record: Name `@`, Value `76.76.21.21`, TTL 300, **Comment left empty**.
   - If that succeeds, **delete** the old A record or any A record pointing at the wrong provider.
   - For **www**: **Add** a new CNAME: Name `www`, Value `cname.vercel-dns.com`, TTL 300. Then remove any old `www` CNAME or A record that points anywhere else.

2. **Strict typing**
   - For **Name**, use exactly `@` (root) or `www` (no spaces or extra characters).
   - For **Value**, type the IP or hostname **by hand** (no copy-paste) to avoid hidden characters.
   - Leave **Comment** blank.

3. **If Add also fails**
   - Try another browser or an incognito/private window.
   - If validation still fails, contact **Vercel support** (dashboard help or support ticket) and cite the exact error when editing DNS for a domain on Vercel nameservers.

### Confirm traffic goes to Vercel

1. **DNS**
   - Run:
     ```bash
     dig signalscoutradar.com +short
     dig www.signalscoutradar.com +short
     ```
   - You should see `76.76.21.21` for the apex/root domain and a Vercel hostname for `www` (such as `cname.vercel-dns.com`), **not** a hostname ending in `cloudfront.net` and not an old non-Vercel IP.

2. **Response headers**
   - Open the live site, trigger the action that was failing (e.g. submit login).
   - In DevTools → **Network**, select the failing request and check **Response headers**.
   - If the response was from CloudFront, you’ll see headers such as `x-amz-cf-id` or `via: CloudFront`. After the fix, those should be gone and the request should succeed or return an application error (e.g. invalid credentials), not a CloudFront 403 page.

3. **Smoke test**
   - Load **https://signalscoutradar.com** (or **https://www.signalscoutradar.com**).
   - Open **Log in** and submit the form. You should get either the dashboard or a login error from the app, **not** the CloudFront 403 page.
   - If the apex/root host works but `www` still shows CloudFront, the `www` DNS record is still wrong or still propagating.

Allow 5–15 minutes (or up to 48 hours in rare cases) for DNS propagation after changing records.
